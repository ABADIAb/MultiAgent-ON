# Orchestrator Script & Literature Review

This report summarizes the findings from the baseline Orchestrator codebase provided by the professor, as well as an analysis of two critical pieces of literature: the ECOC 2024 paper detailing the orchestrator's design, and a comparative paper on RAG Memory Management in Claude Code Agent.

## 1. Literature Review

### 1.1 ECOC 2024: Open Implementation of a Large Language Model Pipeline for Automated Configuration of Software-Defined Optical Networks
- **Core Concept**: The paper presents a natural-language interface for an SDON testbed. To mitigate hallucinations and guarantee valid outputs, it employs a **Compositional Learning** approach.
- **Pipeline Phases**:
  1. **Planning**: Translates open-ended human intent into a structured list of atomic tasks (JSON).
  2. **Execution**: Maps each task to a specific JSON schema (e.g., Lightpath, Service, Measurement) and generates the precise JSON payload required for the Northbound Interface.
  3. **Error Handling**: A feedback loop that catches SDON controller errors (e.g., "ID taken") and re-prompts the LLM to fix them.
- **Constrained Generation**: The most critical technical takeaway is the use of GBNF (Grammar-Based) generation via `llama_cpp`, forcing the LLM to output valid JSON matching a predefined schema. This reduced their error rate from 92% (naive prompting) to 4%.

### 1.2 From Paper to Product: RAG Mapping in Claude Code Agent
This translated paper provides a pragmatic engineering perspective on how commercial coding agents implement memory and retrieval (RAG) compared to academic models.
- **Asynchronous & Decoupled Retrieval**: Instead of forcing the main LLM to synchronously pause and retrieve (like Self-RAG), agents use asynchronous prefetching via a smaller, independent bypass model to reduce latency.
- **Physical vs Semantic Routing**: Agents decide when to use tools based on physical constraints (token context limits) rather than semantic complexity (Adaptive-RAG).
- **Model-Autonomous Multi-Hop**: System-enforced multi-hop reasoning (IRCoT) is replaced by the LLM autonomously deciding to call search tools multiple times. 
- **The Enemy of Multi-Hop**: Aggressive session compaction (summarization) destroys the precise entities needed for iterative multi-hop retrieval.

#### Comparison: Claude Agent RAG vs Hybrid Memory Architecture
Our [[Hybrid_Memory_Architecture]] addresses the exact pitfalls highlighted in the Claude Code Agent paper through the **Tri-Partite Memory System**:
1. **The Compaction Problem**: The paper notes that summarizing context destroys the fine-grained state (like specific IDs or file paths) needed for multi-hop reasoning. Our architecture solves this by offloading physical state tracking to the **Knowledge Graph (Pillar 2)**. The graph is immune to token-summarization loss, guaranteeing deterministic multi-hop reasoning.
2. **Asynchronous Prefetching**: Our **Hierarchical Wikis (Pillar 1)** act similarly to the paper's prefetching phase. The agent loads ground-truth instructions at boot time, bypassing the need for a secondary LLM to guess what rules are needed mid-conversation.
3. **Model Autonomous vs System Enforced**: While Claude relies on the LLM to autonomously remember to use tools, our LangGraph orchestrator enforces the workflow at the system level. If an error occurs, the StateGraph automatically routes the agent to the **Vector RAG (Pillar 3)** for episodic recall, combining academic rigor (enforced routing) with engineering pragmatism.

## 2. Orchestrator Codebase Structure

The cloned repository (`ecoc2024-llm-orchestrator`) is a lightweight, pure Python project that heavily utilizes `llama_cpp`.

### Structure:
- **`planning.py`**: Loads a generalized `task_schema.json`. It queries a local Mixtral GGUF model with the user's prompt, forcing the output to match the grammar of a task list.
- **`execution.py`**: Takes the JSON task list generated by the planner. It loops through each task, loads a specific JSON schema (e.g., `lightpath_schema.json`), and queries the LLM again to generate the final data payload.

## 3. Integration Strategy: Can we use this codebase?

### Verdict: No (Not Directly).
The professor's codebase cannot be used as the *foundation* for our project because:
1. **No Agentic Framework**: It uses raw `llama_cpp` script loops. It lacks state management, tool binding, or agent memory. We are building an autonomous MAS using LangGraph.
2. **Local Inference**: It is hardcoded to use local GGUF models, whereas our architecture implies flexible LLM endpoints.

### How we WILL use it (The Pivot):
While we won't use the Python scripts directly, **we must adopt their Pipeline Architecture and JSON Schemas into our LangGraph Nodes** (see [[experiments/Proposal_Orchestrator_Integration]] for the formal proposal).

**Proposed LangGraph Mapping**:
1. **Planning Node**: Replicates `planning.py`. A LangGraph node that takes the user intent and uses LangChain's `with_structured_output` to generate the task list.
2. **Execution Sub-Agents**: Replicates `execution.py`. Specialized sub-agents (e.g., a Lightpath Agent) that take a task and use `with_structured_output` tied to the professor's exact JSON schemas.
3. **Error Handling Edge**: A conditional edge in LangGraph. If the [[tools_wiki/QoT_Tool|QoT Tool]] or Controller returns an error, the graph routes back to the Planning/Execution node with the error appended to the state, exactly mimicking the paper's Step 3.
