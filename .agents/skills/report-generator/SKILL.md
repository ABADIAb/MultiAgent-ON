## When to Use
Use this skill whenever you are tasked with creating, proposing, or updating a Weekly Report, an Issue Report, or planning the session deliverables during `/debrief1`.

## Core Directives

### 1. Identity & Tone
- **First-Person Singular**: ALWAYS write reports and summaries in the first-person singular ("I/my"). NEVER use plural ("we/our", "the team"), even in "One-Sentence Summaries" generated by templates.
- **Academic Professionalism**: Use a mature, structured, and highly objective tone.

### 2. Report Continuity (The Narrative Arc)
- **Mandatory Previous Report Review**: Before generating a new report, you MUST read the most recent previous report of the same type in the Wiki (e.g., `docs/LLM_Wiki/wiki/weekly_reports/` or `docs/LLM_Wiki/wiki/issues/`).
- **Traceability**: In the Weekly Report's "Planned Goals for This Week", you must explicitly state what was carried forward from the previous week. If a planned goal was not achieved, explicitly state why and whether it is deferred or dropped in the "Actual Progress" section.
- **Issue Continuity**: Move "Pending Issues" from the previous issue report to "Solved Issues" if resolved, or keep them in "Pending" with an updated status.

### 3. Content Granularity & Specificity
- **Technical Specificity**: Avoid generic filler (e.g., "delivered code"). State the exact blocker, architectural decision, or achievement (e.g., "Pivoted to Option 2: Pure Python Physics Port", "Designed V2 LangGraph Orchestrator").
- **Logistical Tracking**: Explicitly track secured logistical prerequisites and remaining blockers (e.g., "VPN access secured", "Testbed SSH connection failing").

### 4. Issue Report Structure
When formatting an Issue Report, you must strictly follow the required markdown template structure. Each issue must clearly separate the debugging process into:
1. **Issue:** (What is broken or pending)
2. **What has already been tried:** (Actions taken, scripts run, errors encountered)
3. **Result:** (What happened after the trial)
4. **Estimated possible solution:** (The proposed next steps to resolve the blocker)

### 5. Final Checklist
Ensure every report concludes with the proper template sections (e.g., "Do You Need Support?", "One-Sentence Summary", "Self-Check Before Submission").
